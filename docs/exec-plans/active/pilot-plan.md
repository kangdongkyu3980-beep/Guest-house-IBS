# 파일럿 계획서: 모바일 연동 검증

## 목적

현재 IBS 운영 HTML 시스템에서, 모바일 사용자가 실제 업무 흐름을 따라가며 `저장 -> 시트 반영 -> 다시 읽기`가 안정적으로 이어지는지 작은 범위로 먼저 증명한다.

이 문서는 채팅이 아니라 저장소 안에 남는 실행 기준이다.

## 예상 효과

1. 계획 문서가 채팅 안이 아니라 저장소 안에 남는다.
2. 파일럿 범위가 분명해진다.
3. 검증과 설계가 같이 보인다.
4. 다음 세션이 계획 문서에서 다시 시작할 수 있다.

## 이번 작업 4항목

### 목표

이번 작업이 끝났을 때는 모바일에서 사용하는 핵심 운영 흐름이 `저장 -> 시트 반영 -> 재조회`까지 신뢰 가능하게 보이고, 운영자가 한 번의 테스트로 연동 상태를 확인할 수 있어야 한다.

### 문맥

지금 제품은 실제 운영에 쓰이는 가치가 이미 있지만, 핵심 로직이 `index.html` 하나에 몰려 있고 로컬 저장과 시트 동기화가 함께 얽혀 있어 데이터 신뢰를 설명하기가 약하다.  
기회는 분명하다. 이미 모바일 웹앱 형태와 실제 업무 흐름은 갖춰져 있으므로, 지금 단계에서 가장 큰 가치는 새 기능 추가보다 `운영 안정성`과 `연동 신뢰`를 먼저 증명하는 것이다.

### 제약

이번에는 아래를 하지 않는다.

1. 로그인, 권한, 테넌트, 결제 같은 SaaS 상위 단계 구현
2. Apps Script를 다른 백엔드로 전면 교체
3. 대규모 UI 재설계
4. 구조 설명 없이 `index.html`에 큰 기능을 더 넣는 작업
5. 운영 데이터 구조를 승인 없이 바꾸는 작업

### 완료 조건

아래까지 되면 이번 작업은 끝으로 본다.

1. 모바일에서 핵심 화면과 저장 흐름이 실제로 열린다.
2. 저장 후 시트 재조회 결과가 기대값과 맞는다.
3. `🧪 연동 테스트` 같은 검증 진입점으로 성공/실패를 구분할 수 있다.
4. 실패 시 어디서 멈춰야 하는지와 어디까지 되돌릴지 문서로 남아 있다.
5. 다음 세션이 이 문서와 `docs/continuity/current.md`만 읽고 바로 이어갈 수 있다.

## 파일럿 범위

### 포함

- 루트 `index.html`의 핵심 모바일 경로
- `🧪 연동 테스트`를 통한 저장 검증
- 로컬 저장 후 시트 재조회 결과 확인
- 개요, 룸맵, 예약 저장, 동기화 상태 확인

### 제외

- 전체 SaaS 전환
- 로그인, 권한, 청구, 테넌시
- 대규모 UI 재설계
- Apps Script 전면 교체
- 회사 업무 범위를 벗어난 추가 기능

## 먼저 증명할 것

파일럿의 첫 번째 증명은 아래 순서로 둔다.

1. 모바일에서 화면이 열리고 주요 탭이 접근된다.
2. 예약 또는 상태 변경이 저장된다.
3. 저장 직후 시트 쪽 결과를 다시 읽어도 데이터가 맞는다.
4. 연동 테스트가 실패/성공을 구분해서 보여준다.

## 어디까지 갈지

이 파일럿은 `실제 운영자가 한 번의 테스트로 연동 신뢰도를 확인할 수 있는 수준`까지 간다.

즉, 다음이 가능하면 충분하다.

- 모바일에서 진입
- 최소 한 번의 저장 수행
- 저장 결과 재확인
- 결과가 성공/실패로 구분됨

## 어디서 멈출지

다음 중 하나에 도달하면 파일럿은 멈춘다.

1. 파일럿 성공 기준을 충족했다.
2. 저장과 재조회 사이의 불일치가 반복된다.
3. 모바일 핵심 흐름이 깨진다.
4. 운영 데이터에 위험이 생긴다.

멈춤은 실패가 아니라 다음 결정을 위한 경계다.

## 롤백 경계

파일럿이 실패하면 아래 순서로 되돌린다.

1. 파일럿에서 추가한 테스트 진입점 비활성화
2. 저장 검증 로직을 마지막 안정 버전으로 복원
3. 데이터 구조 변경이 있었다면 그 변경을 되돌림
4. 운영 흐름에 영향이 있으면 파일럿 자체를 중단

롤백의 기준은 `운영 데이터 안전 > 파일럿 편의 > 기능 확장` 순서다.

## 성공 기준

아래가 모두 만족되면 파일럿 성공으로 본다.

1. 모바일에서 핵심 화면이 열린다.
2. 한 번의 테스트로 저장과 재조회가 연결된다.
3. 로컬과 시트 결과가 기대대로 일치한다.
4. 실패 시 원인이 사용자에게 구분 가능하게 보인다.
5. 다음 세션이 이 문서를 보고 이어서 작업할 수 있다.

## 검증 방법

1. 모바일 또는 모바일 크기 브라우저에서 사이트를 연다.
2. 핵심 경로를 한 번 수행한다.
3. `🧪 연동 테스트`를 실행한다.
4. 저장 결과와 재조회 결과를 비교한다.
5. 성공/실패와 증거를 `docs/continuity/current.md`에 남긴다.

## 설계 메모

- 이 파일럿은 기능 추가가 아니라 신뢰 검증이다.
- 검증 대상은 UI 예쁨이 아니라 데이터 일치성이다.
- 운영자가 설명 없이도 결과를 읽을 수 있어야 한다.
- 파일럿에서 확인된 경계는 나중에 아키텍처 문서로 옮긴다.
- 이 파일럿은 한 번에 많은 것을 증명하지 않고, 가장 위험한 가설부터 작은 범위로 확인한다.
