# SaaS 출시 로드맵: 작은 단계 기준

## 목적

현재 게스트하우스 운영 HTML 시스템을, 실제로 구현 가능하고 출시 가능하며 결제까지 연결되는 SaaS 서비스로 키우기 위한 마일스톤을 작은 검증 단위로 나눈다.

이 문서의 기준은 기능 수가 아니라 위험을 하나씩 줄이는 순서다.

## 전체 원칙

1. 운영 안정성을 잃지 않는다.
2. 각 마일스톤은 하나의 핵심 가설만 먼저 증명한다.
3. 큰 변경 전에 멈춤 조건과 롤백 경계를 같이 적는다.
4. 결제는 제품 신뢰, 권한, 데이터 분리 이후에 붙인다.
5. 큰 마일스톤은 바로 구현하지 말고, 먼저 실행 계획과 작은 검증 단계로 쪼갠다.

## 마일스톤 1: 운영형 MVP 안정화

### 목표

현재 HTML 앱이 실제 운영 도구로 신뢰 가능한 최소 수준에 도달한다.

### 먼저 증명할 것

- 모바일에서 핵심 흐름이 열린다.
- 저장 후 재조회 결과가 맞는다.
- 동기화 실패가 숨지 않는다.

### 완료 기준

- 핵심 화면 진입 가능
- 저장, 재조회, 시트 반영 검증 가능
- 최소 1회 연동 테스트 절차가 문서화됨

### 멈춤 조건

- 데이터 불일치가 반복된다.
- 모바일 핵심 경로가 자주 깨진다.

### 롤백 경계

- 검증용 UI와 동기화 변경만 되돌린다.
- 운영 데이터 구조는 건드리지 않는다.

## 마일스톤 2: 도메인 명세 확정

### 목표

예약, 객실, 게스트, 식사, 체크인/체크아웃 규칙을 문서로 고정한다.

### 먼저 증명할 것

- 핵심 엔티티와 관계를 설명할 수 있다.
- 운영 규칙이 코드 밖에서도 읽힌다.

### 완료 기준

- 엔티티 목록 작성
- 상태 전이와 주요 규칙 정리
- 데이터 모델 설명 문서 존재

### 멈춤 조건

- 용어가 계속 바뀐다.
- 같은 데이터를 서로 다르게 해석한다.

### 롤백 경계

- 문서만 수정한다.
- 코드 구조 변경은 여기서 강행하지 않는다.

## 마일스톤 3: 프론트엔드 경계 분리

### 목표

`index.html`의 결합도를 낮추고, 화면, 상태, 동기화, 도메인 규칙 경계를 분명히 한다.

### 먼저 증명할 것

- 특정 기능 수정 시 영향 범위를 설명할 수 있다.
- 동기화 로직이 화면 로직과 분리 가능하다.

### 완료 기준

- 경계 문서 작성
- 최소 한 영역이라도 분리 전략 수립
- 새 기능 추가 전에 구조 설명 가능

### 멈춤 조건

- 문서 없이 코드 이동만 늘어난다.
- 기존 운영 기능에 회귀가 생긴다.

### 롤백 경계

- 분리 시도는 중단해도 기능은 유지한다.
- UI 동작을 깨는 리팩터링은 되돌린다.

## 마일스톤 4: 백엔드 계약 정의

### 목표

Apps Script 이후를 위한 API 계약을 먼저 문서화한다.

### 먼저 증명할 것

- 프론트가 필요한 읽기/쓰기 계약을 명확히 말할 수 있다.
- 미래 DB 백엔드로 옮겨도 제품 지식이 유지된다.

### 완료 기준

- `booking`, `room`, `guest`, `meal`, `sync status` 계약 초안 작성
- 입력값, 출력값, 오류 응답 정의
- 시트 의존 지점 파악

### 멈춤 조건

- 현재 시트 구현 방식만 그대로 복사하려 한다.
- 고객/운영자 관점 없이 내부 구현만 정의한다.

### 롤백 경계

- 계약 문서 수준에서 되돌린다.
- 운영 API는 승인 없이 바꾸지 않는다.

## 마일스톤 5: 시트 의존 축소

### 목표

프론트가 저장 매체를 직접 알지 않아도 되도록 구조를 만든다.

### 먼저 증명할 것

- 저장 대상을 추상화할 수 있다.
- 시트와 미래 백엔드를 교체 가능하게 만들 수 있다.

### 완료 기준

- 최소 한 저장 경로가 추상화됨
- 프론트가 직접 시트 세부사항을 덜 알게 됨
- 백엔드 교체 비용이 줄어듦

### 멈춤 조건

- 프론트에 시트 전용 분기문이 계속 늘어난다.
- 기존 동기화 안정성이 떨어진다.

### 롤백 경계

- 추상화 레이어만 되돌린다.
- 운영 저장 흐름은 유지한다.

## 마일스톤 6: 내부용 백엔드 1차 구축

### 목표

시트 기반 운영에서 벗어나기 위한 내부용 정식 백엔드를 만든다.

### 먼저 증명할 것

- 최소 한 업무 경로가 시트 없이 저장된다.
- 백엔드가 읽기/쓰기 계약을 안정적으로 지킨다.

### 완료 기준

- 핵심 엔티티 일부 저장 가능
- 기본 배포 환경 존재
- 장애 시 로그 확인 가능

### 멈춤 조건

- 운영 기능보다 인프라 작업만 커진다.
- 읽기보다 쓰기 실패가 많다.

### 롤백 경계

- 내부 백엔드 실험은 중단 가능
- 시트 기반 운영 경로는 마지막 안전망으로 남긴다.

## 마일스톤 7: 인증과 권한

### 목표

운영자 로그인과 최소 권한 구분을 도입한다.

### 먼저 증명할 것

- 사용자 구분이 실제로 동작한다.
- 무권한 접근이 차단된다.

### 완료 기준

- 로그인 가능
- 최소 역할 구분 존재
- 보호된 화면 또는 API가 생김

### 멈춤 조건

- 인증은 붙었지만 권한은 비어 있다.
- 운영자 본인도 정상 작업이 어려워진다.

### 롤백 경계

- 인증 실험은 내부 사용자 범위에서만 적용
- 운영 차질 시 기존 접근 경로를 임시 복구

## 마일스톤 8: 감사 로그와 변경 추적

### 목표

누가 무엇을 바꿨는지 기록할 수 있게 한다.

### 먼저 증명할 것

- 예약 변경 1건의 이력을 추적할 수 있다.
- 장애나 분쟁 시 변경 근거를 제시할 수 있다.

### 완료 기준

- 생성, 수정, 삭제 로그가 남음
- 최소 조회 화면 또는 로그 확인 경로 존재

### 멈춤 조건

- 본 데이터보다 로그 신뢰도가 더 낮다.
- 개인정보 범위가 불명확하다.

### 롤백 경계

- 로그 저장 형식은 조정 가능
- 본 기능 저장 로직과 강하게 결합하지 않는다.

## 마일스톤 9: 단일 고객 정식 운영

### 목표

한 회사 기준으로 시트 의존 없이 운영 가능한 상태를 만든다.

### 먼저 증명할 것

- 실제 업무가 새 구조에서 돌아간다.
- 운영자가 기존보다 불편하지 않다.

### 완료 기준

- 일정 기간 실사용 가능
- 주요 운영 이슈 대응 가능
- 긴급 복구 경로 존재

### 멈춤 조건

- 운영자가 다시 시트로 돌아가야 한다.
- 핵심 업무가 수작업으로 보완된다.

### 롤백 경계

- 단일 고객 범위 안에서만 전환
- 필요 시 시트 운영으로 제한적 복귀

## 마일스톤 10: 멀티테넌시 설계

### 목표

여러 고객이 서로 완전히 분리된 데이터로 사용할 수 있게 설계한다.

### 먼저 증명할 것

- 고객사, 지점, 건물, 사용자 범위를 분리할 수 있다.
- 한 고객의 데이터가 다른 고객에게 보이지 않는다.

### 완료 기준

- 테넌트 모델 정의
- 데이터 경계 정의
- 온보딩에 필요한 기본 설정 정의

### 멈춤 조건

- 고객 구분이 UI 수준에만 머문다.
- 데이터 경계가 문서 없이 추정에 의존한다.

### 롤백 경계

- 멀티테넌시는 설계와 내부 베타 범위에서 검증
- 단일 고객 운영은 계속 유지

## 마일스톤 11: 고객 온보딩 최소 버전

### 목표

코드 수정 없이 신규 고객 1곳을 세팅할 수 있게 만든다.

### 먼저 증명할 것

- 건물, 객실, 기본 옵션 세팅이 가능하다.
- 운영자가 아닌 사람도 기본 설정을 따라갈 수 있다.

### 완료 기준

- 신규 고객 생성 절차 존재
- 초기 설정 체크리스트 존재
- 최소 한 번의 내부 온보딩 완료

### 멈춤 조건

- 여전히 개발자 수작업이 필수다.
- 설정 단계가 문서 없이 설명되지 않는다.

### 롤백 경계

- 온보딩 UI보다 운영용 수동 등록 경로를 유지
- 고객 데이터 생성 실패 시 즉시 중단

## 마일스톤 12: 결제 전 준비

### 목표

무엇에 대해 돈을 받을지 먼저 명확히 한다.

### 먼저 증명할 것

- 과금 단위가 분명하다.
- 무료와 유료 경계가 설명 가능하다.
- 해지, 환불, 갱신 정책이 있다.

### 완료 기준

- 플랜 정의
- 과금 기준 정의
- 결제 이벤트와 서비스 상태 관계 정리

### 멈춤 조건

- 가격만 있고 서비스 경계는 없다.
- 지원, 환불, 해지 정책이 비어 있다.

### 롤백 경계

- 결제 붙이기 전에 정책 문서로만 조정
- 실제 청구는 시작하지 않는다.

## 마일스톤 13: 결제 시스템 연결

### 목표

구독 생성, 결제 성공/실패, 갱신, 해지 흐름을 붙인다.

### 먼저 증명할 것

- 테스트 결제로 가입부터 해지까지 한 사이클이 돈다.
- 결제 상태와 서비스 상태가 일치한다.

### 완료 기준

- 결제 제공자 연동
- 테스트 결제 성공
- 웹훅 또는 상태 반영 로직 존재
- 실패 결제 대응 흐름 존재

### 멈춤 조건

- 결제 성공 후 서비스가 열리지 않는다.
- 해지 후 권한 정리가 안 된다.

### 롤백 경계

- 테스트 모드에서 먼저 검증
- 실제 청구 시작 전 내부 검토 완료

## 마일스톤 14: 베타 출시

### 목표

소수 고객에게만 공개해 실제 사용과 결제까지 검증한다.

### 먼저 증명할 것

- 고객 1~3곳이 온보딩 가능하다.
- 사용과 결제가 실제로 이어진다.

### 완료 기준

- 베타 고객 확보
- 핵심 운영 지원 가능
- 반복되는 장애 패턴 파악

### 멈춤 조건

- 고객 지원 비용이 너무 크다.
- 데이터 분리, 권한, 청구 문제 중 하나라도 치명적이다.

### 롤백 경계

- 베타 고객 수를 제한
- 신규 유입 중단 가능

## 마일스톤 15: 정식 출시

### 목표

지원, 청구, 백업, 장애 대응을 포함한 정식 운영 상태로 전환한다.

### 먼저 증명할 것

- 수동 대응 없이 기본 운영이 가능하다.
- 고객 문의와 청구 흐름을 처리할 수 있다.

### 완료 기준

- 운영 문서 존재
- 백업과 복구 기준 존재
- 지원 절차 존재
- 청구 운영 가능

### 멈춤 조건

- 고객 증가가 운영 체계를 앞지른다.
- 장애 대응 시간이 관리 불가능하다.

### 롤백 경계

- 정식 확장은 멈출 수 있어야 한다.
- 베타 수준 운영으로 제한 복귀 가능해야 한다.

## 추천 순서 요약

1. 지금 앱을 믿을 수 있게 만든다.
2. 데이터와 업무 규칙을 문서화한다.
3. 백엔드와 인증을 붙여 내부 제품으로 만든다.
4. 멀티테넌시와 온보딩을 넣어 SaaS 형태로 바꾼다.
5. 결제를 붙여 판매 가능 상태로 만든다.
