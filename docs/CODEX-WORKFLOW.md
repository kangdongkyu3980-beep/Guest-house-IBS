# CODEX-WORKFLOW.md

## 목적

이 문서는 저장소 안에서 Codex와 일할 때 반복해서 설명하게 되는 작업 방식을 정리한다.

`AGENTS.md`는 짧은 지속 규칙을 둔다.  
이 문서는 그 규칙을 실제 작업 흐름으로 풀어 쓴다.

## 왜 필요한가

Codex는 문맥, 구조, 경계가 분명할수록 더 안정적으로 작동한다.  
반대로 범위가 크고 요청이 흐리면, 결과도 길어지고 수정 비용이 커진다.

그래서 이 저장소에서는 아래 원칙을 따른다.

1. 큰 작업은 먼저 계획으로 자른다.
2. 작업은 가능한 한 이슈처럼 적는다.
3. 결과는 채팅이 아니라 저장소 문서에도 남긴다.
4. 다음 세션이 문서만 읽고 이어갈 수 있어야 한다.

## 권장 작업 흐름

### 1. 시작 전

먼저 아래 문서를 읽는다.

1. `AGENTS.md`
2. `docs/continuity/current.md`
3. 관련 `docs/exec-plans/active/`
4. 필요한 경우 `docs/code-map/`

### 2. 작업 요청 작성

작업 요청은 가능하면 아래 요소를 포함한다.

1. 바꿀 범위
2. 원하는 결과
3. 제외 범위
4. 검증 기준
5. 참고 파일 또는 패턴

좋은 요청 예시는 GitHub 이슈와 비슷하다.

### 3. 큰 작업 다루기

큰 작업은 바로 구현하지 않는다.

아래 순서로 나눈다.

1. 범위 정리
2. 작은 검증 단계 정의
3. 멈춤 조건 정의
4. 롤백 경계 정의
5. 그 다음 구현

이런 작업은 `docs/exec-plans/active/`에 먼저 남긴다.

### 4. 작업 크기 기준

Codex에게 맡길 작업은 가능한 한 아래 범위가 좋다.

1. 한 가지 핵심 목표
2. 몇 개 파일 또는 한 논리 단위
3. 검증 방법이 분명한 변경
4. 실패했을 때 되돌릴 수 있는 범위

너무 큰 작업은 여러 단계로 나눈다.

## 이 저장소에 맞는 적용 방식

### 프론트엔드

- `index.html`은 실제 운영 지식이 많은 핵심 파일이다.
- 큰 기능 추가보다 경계 문서화와 동기화 신뢰 보강이 먼저다.
- 모바일과 저장/재조회 일치 여부를 항상 우선 본다.

### 백엔드 연동

- `integrations/google-apps-script/`는 운영 데이터와 직접 연결된다.
- 승인 없는 운영 변경은 피한다.
- 읽기/쓰기 계약을 먼저 문서화한 뒤 교체를 논의한다.

### 문서

- 채팅에서 끝나는 설명을 줄이고 저장소 안 문서로 옮긴다.
- 현재 상태는 `docs/continuity/current.md`에 남긴다.
- 실행 계획은 `docs/exec-plans/active/`에 남긴다.
- 구조 설명은 `docs/ARCHITECTURE.md`, `docs/code-map/`으로 보낸다.

## Best Practices 취사선택

아래 항목은 이 저장소에 적극 반영한다.

1. 큰 변경 전 계획 먼저 세우기
2. 작업 요청을 이슈처럼 구조화하기
3. 지속 규칙은 `AGENTS.md`에 두기
4. 다음 세션 재시작이 가능하도록 현재 상태 남기기
5. 작은 작업 단위로 검증하며 확장하기

아래 항목은 지금은 제한적으로만 쓴다.

1. 자동화된 대량 병렬 작업
2. 대규모 멀티 PR 흐름
3. 환경 설정 자동화에 큰 투자

이유는 현재 저장소가 아직 단일 HTML 중심 구조이고, 운영 안정화가 더 우선이기 때문이다.

## 세션 종료 체크

의미 있는 작업을 했다면 아래 중 하나 이상을 갱신한다.

1. `docs/continuity/current.md`
2. 관련 `docs/exec-plans/active/`
3. `docs/code-map/`
4. 필요한 기준 문서
