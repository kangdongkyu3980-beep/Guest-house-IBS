# AGENTS.md

## 목적

이 문서는 프롬프트에서 매번 반복하던 규칙을 지속 지침으로 옮겨 두는 루트 규칙 문서다.

여기에는 오래 유지할 기준만 둔다.  
오늘만 유효한 작업 지시, 제품 기획 전체, 장문의 배경 설명은 넣지 않는다.  
자세한 설명은 `docs/`로 연결한다.

## 적용 범위

- 루트 규칙은 저장소 전체에 적용한다.
- 단, 하위 디렉터리에 별도 `AGENTS.md`가 있으면 그 범위에서는 하위 규칙을 우선한다.
- 현재 프론트엔드 실행 파일이 루트에 있으므로, 루트 규칙은 공통 규칙이면서 프론트엔드 기본 규칙을 함께 포함한다.

## 먼저 읽을 문서

1. 현재 상태와 재시작 지점: `docs/continuity/current.md`
2. 전체 구조와 경계: `docs/ARCHITECTURE.md`
3. 프론트엔드 기준: `docs/FRONTEND.md`
4. 신뢰성 기준: `docs/RELIABILITY.md`
5. 보안 기준: `docs/SECURITY.md`
6. 현재 실행 계획: `docs/exec-plans/active/`

## 공통 작업 원칙

1. 테스트 우선을 기본값으로 둔다.
2. 큰 변경 전에 계획을 먼저 남긴다.
3. 승인 없는 운영 변경은 피한다.
4. 운영 중인 실제 흐름을 먼저 보존한다.
5. 코드 변경과 문서 변경이 함께 있어야 구조 변경이 완료된다.
6. 문서 규칙은 별도 `docs/AGENTS.md`로 나누지 않고, 루트 `AGENTS.md`와 `docs/` 문서 본문에서 관리한다.
7. 큰 작업은 바로 코딩부터 하지 말고, 먼저 범위와 영향도를 정리한 뒤 진행한다.
8. 작업 요청은 가능한 한 GitHub 이슈처럼 쓴다.
9. 한 번에 너무 큰 작업보다, 검증 가능한 작은 단계로 나눈 작업을 우선한다.

## 공통 검증 규칙

모든 변경은 아래를 먼저 확인한다.

1. 첫 진입 시 치명적 오류 없이 열린다.
2. 실패 시 전체 화면이나 전체 흐름이 멈추지 않는다.
3. 변경 전후 핵심 사용자 흐름이 유지된다.
4. 테스트 또는 검증 방법이 설명 가능하다.
5. 운영 데이터, 비밀값, 외부 연동 URL을 무심코 바꾸지 않는다.
6. 다음 세션이 문서만 읽고 다시 시작할 수 있어야 한다.

## 작업 단위 기준

Codex 작업은 아래 기준으로 자른다.

1. 가능하면 한 번의 작업은 한 가지 핵심 목표만 가진다.
2. 큰 기능은 바로 구현하지 말고 `계획 -> 작은 검증 -> 확장` 순서로 나눈다.
3. 한 작업은 대체로 짧은 설명, 명확한 범위, 검증 방법을 함께 가져야 한다.
4. 범위가 커지면 먼저 `docs/exec-plans/active/`에 실행 계획을 남긴다.

## 요청 작성 기준

작업 요청은 가능한 한 이슈처럼 적는다.

1. 바꾸려는 파일이나 영역
2. 원하는 결과
3. 하지 말아야 할 것
4. 검증 기준
5. 참고 문서나 기존 패턴

설명이 길어지면 채팅에만 두지 말고 `docs/exec-plans/active/` 문서로 남긴다.

## 프론트엔드 기본 규칙

루트 범위의 프론트엔드 작업은 아래를 추가로 따른다.

1. 모바일에서 핵심 경로를 우선 검증한다.
2. 저장, 재조회, 상태 반영이 서로 맞는지 확인한다.
3. 서비스 워커와 캐시 영향 여부를 항상 검토한다.
4. 큰 UI 변경 전에는 흐름도나 체크리스트 수준의 계획을 남긴다.

## 승인과 제외 기준

승인 가능한 변경:

- 기존 핵심 운영 흐름을 유지한다.
- 검증 방법이 있다.
- 필요한 문서가 함께 갱신되었다.
- 승인 없이 운영값이나 실서버 동작을 바꾸지 않는다.

루트 문서에서 제외할 내용:

- 오늘만 유효한 작업 지시
- 제품 기획 전체
- 취향 중심 규칙
- formatter, linter, CI가 이미 강제하는 항목
- 긴 구현 설명과 긴 예외 목록

## docs로 분리할 기준

아래 중 하나에 해당하면 `AGENTS.md`에 길게 쓰지 말고 `docs/`로 분리한다.

1. 배경 설명이 5줄 이상 필요하다.
2. 예외 케이스가 여러 개라서 표나 사례가 필요하다.
3. 아키텍처, 보안, 신뢰성처럼 한 번에 판단하기 어려운 주제다.
4. 실행 순서, 마이그레이션, 릴리스 단계처럼 살아 있는 계획이다.
5. 코드 맵, 도메인 모델, 장애 대응처럼 다음 세션에도 다시 읽어야 한다.

## 문서 작업 규칙

1. `docs/`에는 설명 문서를 두고, 별도 규칙 파일은 두지 않는다.
2. 같은 내용을 여러 문서에 길게 반복하지 않는다.
3. 구조, 검증, 보안, 실행 계획은 성격에 맞는 문서로 분리한다.
4. 다음 세션의 사람이 읽고 바로 이어갈 수 있게 현재 판단 근거와 다음 행동을 남긴다.

## 문서 위치

- 전체 구조: `docs/ARCHITECTURE.md`
- 프론트엔드 기준: `docs/FRONTEND.md`
- 신뢰성 기준: `docs/RELIABILITY.md`
- 보안 기준: `docs/SECURITY.md`
- Codex 작업 방식: `docs/CODEX-WORKFLOW.md`
- 코드 맵: `docs/code-map/`
- 현재 상태와 이어하기: `docs/continuity/current.md`
- 활성 실행 계획: `docs/exec-plans/active/`
- 완료된 계획 기록: `docs/exec-plans/completed/`

## 세션 종료 규칙

의미 있는 변경이 있었다면 최소 하나 이상 갱신한다.

1. 상태가 바뀌었으면 `docs/continuity/current.md`
2. 구조 이해가 바뀌었으면 `docs/ARCHITECTURE.md` 또는 `docs/code-map/`
3. 검증 기준이 바뀌었으면 `docs/RELIABILITY.md`
4. 우선순위가 바뀌었으면 `docs/exec-plans/active/`
